# AR巡检项目梳理（完整版）

> 本文档基于现存代码与我的项目回忆联合还原，重点保留真实做法、真实难点和对应解决方案，方便后续写简历、面试复盘和项目讲解。

## 一、项目定位

这个项目本质上是一个 **AR眼镜 + Android端 + Django后端** 的巡检系统。

核心目标不是单纯“拍照上传”，而是把完整的巡检流程真正跑通：
- 后端下发任务；
- 巡检人员佩戴雷鸟 AR 眼镜执行任务；
- 到达巡检点后通过扫码识别当前位置；
- 在当前位置拍照并上传；
- 多个位置重复该流程；
- 最后提交任务；
- 后端将任务改为完成状态，并按位置整理图片，生成巡检结果展示。

这个项目表面看流程不复杂，但真正难的地方并不只是增删改查，而是：
1. **AR眼镜不是普通安卓手机，交互方式和显示方式都不一样**；
2. **前后端通信在局域网环境下很容易出问题**；
3. **图片上传时必须和具体位置绑定，否则报告一定会乱**；
4. **任务流要闭环**，不能只是“传张图就结束”。

## 三、系统架构梳理

## 1. 总体架构

整个系统可以理解为三层：

### 第一层：AR 眼镜 Android 客户端
负责：
- 展示界面；
- 接收任务；
- 调用 AR SDK 处理交互；
- 打开相机 / 扫码；
- 拍照上传；
- 提交任务完成状态。

### 第二层：Django 服务端
负责：
- 存储任务信息；
- 存储图片、语音文件；
- 提供任务查询接口；
- 提供上传接口；
- 维护任务状态；
- 提供管理页面查看任务、图片、语音。

### 第三层：数据存储层
- 数据库存储任务记录、图片记录、语音记录；
- 媒体文件存储在 `media/` 目录下；
- 后端通过 Django 的 `MEDIA_URL` / `MEDIA_ROOT` 对上传文件做统一管理。

## 五、完整业务流程还原

结合代码和你的回忆，这个项目最合理、最完整的业务流程应该是下面这样：

### 第一步：后台创建任务
管理端先在 Django 后台或网页端创建任务：
- 任务内容 `task`
- 执行人 `name`
- 初始状态 `state = 未完成`

对应页面 / 接口：
- `ar巡检/django-project-master/templates/missions.html:9`
- `ar巡检/django-project-master/mission/views.py:20`

### 第二步：眼镜端连接后端并获取任务
眼镜端启动后，先与 Django 后端联通，再按执行人查询任务。

现存后端里有一个很关键的接口：
- `ar巡检/django-project-master/mission/views.py:91`
- `mission_by_name(request, name)`

对应 URL：
- `ar巡检/django-project-master/mission/urls.py:21`

这个接口说明你的系统已经考虑到了：
- **按人查任务**；
- 前端只拿和当前巡检员相关的任务，而不是把所有任务都拉下来。

### 第三步：到达巡检点后扫码识别位置
这一环节是整个巡检流里最关键的一环。

你的做法不是拍完照片再猜测它属于哪个位置，而是：
- 每个巡检点都有二维码；
- 二维码中包含该位置的业务信息，比如“咖啡机”“配电箱”“控制柜”等；
- 眼镜端先扫码拿到位置，再进入拍照 / 上传流程。

这样设计的好处是：
- 位置来源明确；
- 无需后处理猜测；
- 一张图和巡检点的关系在上传时就已经确定。

### 第四步：拍照并上传，同时携带位置信息
这一步是你这个项目里一个非常容易被忽略、但实际上最重要的设计点。

如果只是普通流程：
- 扫码；
- 拍照；
- 自动上传；

那么一旦出现下面这种场景，数据就会乱：
- 同一个位置拍多张图；
- 同一个设备拍不同角度；
- 同一个点位补拍多次；
- 执行人中途回到上一个点重拍。

这时如果上传记录里不带“位置”，那后端最终只会得到一堆图片文件，根本不知道它们分别对应哪个巡检点。

所以你实际设计的关键点是：
- **二维码中带位置信息**；
- **图片上传时，位置信息和图片一起上传**；
- 后端保存图片记录时，把这个位置标识一并落库。

从现存 Django 代码看，图片上传接口为：
- `ar巡检/django-project-master/photo/views.py:36`

它接收：
- `name`
- `image_file`

而 `Image` 模型里也有：
- `name`
- `time_code`
- `image_file`

也就是说，当前代码层面的数据结构天然支持你这个设计：
- `name` 可以承载位置标识；
- `time_code` 记录上传时间；
- `image_file` 记录图片本体。

这正是你后来做“按位置整理报告”的基础。

### 第五步：重复巡检多个点位
完成一个点位后，继续下一处：
- 扫码识别位置；
- 拍照；
- 位置 + 图片一起上传；
- 保存记录。

整个任务过程中，系统积累的是 **结构化的巡检记录**，而不是零散图片。

### 第六步：提交任务并结束任务流
巡检结束后，用户不是直接退出，而是需要明确“提交任务”。

这一点很重要，因为：
- 没有结束动作，后端不知道任务什么时候真正完成；
- 没有结束动作，就不能生成最终结果；
- 没有状态切换，管理端看不到任务进度。

后端当前 `Mission` 模型本身就有状态字段：
- `state`

并且前端表单里已经定义了：
- `未完成`
- `进行中`
- `已完成`

对应文件：
- `ar巡检/django-project-master/templates/missions.html:20`

所以从业务设计上，你这个任务是有明确状态流转的：
- 未完成 → 进行中 → 已完成

### 第七步：生成巡检结果 / 报告展示
严格说，当前保留下来的 Django 代码里，我没有看到一个独立的“报告生成页面”模块，但从你描述的实际设计来看，报告逻辑的核心并不在页面花样，而在 **数据组织方式是否正确**。

你真正做对的地方是：
- 扫码得到位置；
- 上传时图片携带位置；
- 任务完成后按位置聚合图片；
- 同一位置的多张图会被整理在同一组里展示。

所以这个项目的“报告生成”本质上是：
- 以任务为维度收拢记录；
- 以位置为维度聚合图片；
- 输出一个结构化的巡检结果。

也就是说，难点不在“做个页面”，而在 **前面数据结构设计是否一开始就正确**。

## 六、这个项目最真实的技术难点与解决方案

下面这部分是面试里最值得讲的，也是这个项目真正有含金量的地方。

## 难点 1：AR 眼镜不是普通安卓终端，显示方式必须重做

### 问题本质
普通安卓 App 默认是：
- 一个屏幕；
- 一套触控逻辑；
- 一个全屏 UI。

但 AR 眼镜不是这种模式。

你当时遇到的问题是：
- 眼镜是双目显示，不是普通手机单屏；
- 如果直接沿用手机布局，佩戴体验不对；
- 需要把显示内容拆成适合双目 / 双区域观看的形式；
- 还要兼顾状态信息展示和相机预览。

### 你的解决方案
你实际做的思路是：
- 基于官方 RayNeo Demo 改，而不是自己硬造底层；
- 保留 SDK 的眼镜交互能力；
- 把页面布局改造成双区域结构；
- 一部分区域显示相机预览，另一部分区域显示状态信息 / 任务信息；
- 在视觉上适配眼镜端的观看方式。

从现存代码也能看出来：
- `activity_camera_preview.xml` 把页面拆成了左右两块；
- 标题区域保留在底部；
- 不再是原版 Demo 那种单纯的按钮菜单页。

### 为什么这能解决问题
因为这不是单纯“把页面调好看”，而是把：
- 显示空间分工；
- 眼镜视野阅读习惯；
- 相机信息和任务信息的并行展示；
统一到了一套适合眼镜端的 UI 结构里。

这个点很适合面试时概括为：
> AR 眼镜不是普通手机，最开始要先解决显示范式问题。我把官方 Demo 的单纯示例界面改造成更适合眼镜端使用的双区域界面，把预览区和信息区拆开，才能让后面的扫码、任务显示、状态反馈都正常工作。

## 难点 2：RayNeo SDK 的交互方式和普通安卓不同

### 问题本质
普通安卓开发里，很多时候就是：
- `setOnClickListener`
- 触屏点击
- 页面跳转

但眼镜端不是纯触屏交互。

你这里需要按 RayNeo SDK 的方式去接管输入事件，否则：
- 焦点不对；
- 操作逻辑不对；
- 页面虽然能显示，但不能正常用。

### 你的解决方案
你采用的是 SDK 自带的一套交互框架：
- `TempleActionViewModel`
- `FocusHolder`
- `FixPosFocusTracker`
- `watchAction(...)`

在 `TitleView.kt` 里还能看到：
- 双击结束页面；
- 焦点切换；
- 左右侧效果处理；
- 3D 效果处理。

### 为什么这能解决问题
因为你不是把眼镜当成普通安卓设备，而是顺着 SDK 的交互模型去写。这样做的结果是：
- 焦点可控；
- 操作可控；
- 页面切换和事件响应符合眼镜设备的使用方式。

面试时可以说：
> 这个项目里安卓端最开始的坑，不是业务逻辑，而是设备交互模型不一样。普通安卓是点击事件为主，但 RayNeo 眼镜要按 SDK 的事件体系来写，我后来是顺着它的焦点和动作模型去重构交互，界面才真正可用。

## 难点 3：扫码 / 相机接入不是普通手机相机，视角和方向需要重新适配

### 问题本质
你这里并不是手机上直接调一个扫码库就结束了。

在 AR 眼镜场景下，常见问题包括：
- 相机方向与显示方向不一致；
- 横屏状态下预览和真实视野不一致；
- 扫码窗口和用户看到的画面可能有偏差；
- 必须做额外的方向修正，否则体验很差。

你自己回忆里提到的一个典型点就是：
- **相机画面需要旋转 90° 之后视野才正常**。

### 你的解决方案
- 接入 CameraX 做相机预览；
- 使用后置相机；
- 在横屏眼镜场景下重新调整预览逻辑；
- 对扫码使用的画面方向做适配，让用户看到的画面与扫码识别逻辑尽量一致。

现存代码中能确认的是：
- 已经接入 CameraX；
- 已单独实现 `CameraPreviewActivity`；
- 预览长宽比设置成更适合眼镜的 `16:9`。

### 为什么这能解决问题
因为巡检项目里的扫码不是附属功能，而是整个“位置识别”的入口。如果这里方向不对、视野不对，后面的位置信息就拿不到，整个巡检链路就断了。

面试表达建议：
> 扫码这一块难点不在调用接口，而在 AR 眼镜的相机和普通手机不一样，预览方向、横屏显示和实际视野都要重新适配。我当时专门处理过相机方向问题，不然扫码体验会很差，位置识别也不稳定。

> 说明：现存代码里保留了 CameraX 预览页，但完整扫码实现没有完整保留下来，因此这里是“代码 + 项目回忆”的联合还原。

## 难点 6：整个项目里最容易被忽略、但最关键的难点，是“图片与位置的绑定”

这是这个项目里我认为最值得重点讲的部分。

### 为什么这是核心难点
很多人做这种巡检系统时，会把流程想得太简单：
- 到一个位置；
- 扫码；
- 拍照；
- 上传。

看起来没毛病，但一旦进入真实使用场景，马上就会乱。

比如：
- 同一个位置拍 3 张图；
- 同一个设备正面一张、侧面一张、细节一张；
- 先拍了 A 点，后来又回到 A 点补拍；
- B 点和 C 点连续拍摄，网络稍有延迟就可能打乱顺序。

如果系统里没有一个 **稳定、可追踪的位置标识**，那最终报告出来就是一堆图片，完全不知道谁对应谁。

### 你的解决方案
你的设计思路其实很正确，而且很工程化：
- **二维码中直接写入巡检位置标识**；
- **扫码先拿位置，再允许拍照上传**；
- **上传时把图片和位置一起提交到后端**；
- **后端按位置聚合图片，而不是按单张上传顺序拼接**。

### 为什么这个方案有效
因为它解决的是“数据结构正确性”问题，而不只是“流程能跑”。

上传顺序可以错；
网络可以慢；
同一位置可以有多张图；
但只要每张图都带着同一个位置标识，后端最终都能把它们重新整理到一起。

换句话说：
- 你不是用“上传顺序”当关系；
- 你是用“位置标识”当关系。

这就是为什么最后报告不会乱。

### 这块怎么讲最能体现你的思考深度
你可以这样说：
> 这个项目里有一个很容易被忽略的设计难点，就是图片和巡检位置的绑定。如果只是扫码后直接拍照上传，实际使用时同一个位置往往会拍多张图，或者补拍，最后报告一定会乱。所以我当时把二维码里的位置信息直接作为上传数据的一部分，让每张图从进入后端那一刻起就带着位置标签。这样任务结束后，系统就能按位置把同一处设备的多张图自动归并出来。

这段话是这个项目里最加分的地方之一。

## 难点 7：任务必须有结束机制，否则无法形成闭环

### 问题本质
很多项目做到上传就停了，但巡检系统不能这样。

因为巡检不是上传几张图就结束，而是要回答几个问题：
- 任务是否真的完成？
- 是否还在执行中？
- 当前这批上传记录属于哪个任务？
- 什么时候可以出最终结果？

### 你的解决方案
你当时把流程设计成：
- 领取 / 获取任务；
- 巡检中逐点扫码与拍照；
- 全部完成后手动提交任务；
- 后端把该任务状态改成“已完成”；
- 再基于该任务下的记录整理结果。

### 为什么这很重要
因为只有这样，系统才不是“文件上传器”，而是完整的巡检系统。

有了任务结束动作之后：
- 管理端可以知道任务什么时候完成；
- 图片、音频、位置记录可以和任务状态对应起来；
- 报告整理才有明确触发时机。

面试时可以概括为：
> 我当时除了做上传接口，更重要的是把整个任务流设计成了闭环。因为巡检系统必须知道任务何时结束，只有任务进入完成状态之后，系统才能把对应的位置和图片整理成最终结果。
